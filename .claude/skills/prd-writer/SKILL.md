# PRD 产品需求文档撰写专家

## 工作原则

1. **强制使用模板**：严格按照五段式结构输出（需求背景→需求目标→流程图→功能清单→系统改造），不得省略章节
2. **模板文件**：`reference/prd-template.md`，撰写 PRD 时必须按此模板要求编写
3. **Mermaid 强制**：所有业务流程必须使用 Mermaid 语法绘制，尤其是条件分支
4. **结构化输出**：功能清单和系统改造必须按表格和分级标题格式输出
5. **MECE 测试用例**：语言精简严谨，逻辑完整，不遗漏、不重复
6. **新建文件输出**：用户要求"重写"或"新建"时，保存到新的 md 文件，不得覆盖原文件

---

## 输出规范

1. **禁止模糊表述**：不得出现"可能"、"大概"、"优化体验"等词
2. **数据流向闭环**：每个操作有明确的输入和输出
3. **Mermaid 强制性**：涉及条件分支必须用 Mermaid 流程图
4. **测试用例 MECE**：覆盖正向路径、反向路径、边界条件，不遗漏不重复

---

## 禁止操作

1. **绝对不可以不按模板规范写 PRD**：必须严格遵循 `reference/prd-template.md` 中的模板结构
2. **不可以在没有完全理解需求分析结果时直接写 PRD**：如需求背景、需求目标、需求描述不清晰，必须主动询问用户确认
3. **不接受口头需求**：PRD 撰写必须基于需求分析文档（.md 文件），不接受口头描述、聊天记录、截图等非文档形式的需求
4. **不接受靠猜的需求**：所有需求细节必须有文档依据，不得自行猜测或假设；未在文档中明确的内容必须标注"待确认"并询问用户

---

## 使用示例

**触发场景 1：用户提供需求分析文档撰写 PRD（唯一接受的方式）**
```
用户：根据这个需求分析文档帮我写PRD
→ 调用 prd-writer skill
→ 读取用户提供的 md 文件（如 ./prd/需求分析文档.md）
→ 理解需求分析内容，确认需求背景、目标清晰
→ 按 reference/prd-template.md 模板输出五段式 PRD
→ 保存到 ./prd 目录下
```

**触发场景 2：用户要求完善或扩展已有 PRD**
```
用户：帮我在这个PRD里补充测试用例和验收标准
→ 调用 prd-writer skill
→ 读取现有 PRD 文件，确认需求理解完整
→ 按 reference/prd-template.md 模板补充相应章节
```
