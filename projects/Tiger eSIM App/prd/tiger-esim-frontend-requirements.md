#### 2.2 套餐类型两级切换（P1）

**现状问题：** 流量档位（500MB/1GB/3GB/无限量）以四格网格呈现，与天数列表并列排布，信息层级不清晰；通话/短信作为「加购」附加项，用户感知弱。

**改造方案：** 参考 Holafly 等主流 eSIM 应用，将套餐选购重构为**两级切换**结构，优先展示套餐类型维度，让用户先明确自己的使用场景，再选择具体规格。

**第一级：主 Tab（下划线样式）**

```
[  纯数据  ]  [  数据+通话+短信  ]
─────────────
```

- 两个 Tab 并排，选中项下方显示品牌色下划线
- 默认选中「纯数据」
- 切换时，第二级 Toggle 和天数列表同步更新

**第二级：套餐档位 Toggle（胶囊样式，仅在「纯数据」下显示）**

```
┌──────────────────────────────┐
│  [ 无限量 ]   [ 固定流量 ]   │
└──────────────────────────────┘
```

- 胶囊形双选，选中侧为品牌色填充，未选侧为灰底
- 默认选中「无限量」
- 切换至「固定流量」后，天数列表展示多个流量档位卡片（1GB / 3GB / 5GB）

**天数列表联动规则：**

| 主 Tab | 子 Toggle | 天数卡片展示内容 |
|--------|-----------|----------------|
| 纯数据 | 无限量 | 「无限 GB」+ 价格 |
| 纯数据 | 固定流量 | 「1 GB」「3 GB」「5 GB」各一行 + 对应价格 |
| 数据+通话+短信 | 不显示 | 「X GB · Y 分钟 · Y SMS」+ 价格（捆绑套餐） |

**说明：**
- 选中「数据+通话+短信」时，通话和短信已内置在套餐内，不再展示「加购选项」区域
- 选中「纯数据」时，「加购选项」区域（短信包 / 通话分钟包）正常显示

#### 2.4 套餐详情页 + 购买流程合并（P1）

**现状问题：** 首次购买需经历 7 步——首页 → 地区筛选 → 套餐列表 → 套餐详情页 → 购买确认页 → 支付方式 → 支付。其中「套餐详情页」和「购买确认页」内容高度重叠（均展示套餐名称、价格、有效期），用户需要看两次相同信息，产生额外摩擦。

**优化方案：** 将「购买确认页」的核心内容（支付方式选择 + 总价确认 + 优惠码）下沉至套餐详情页底部，通过一个吸底的「立即购买」操作区实现，用户无需跳转即可完成确认并支付。**目标：首购路径从 7 步压缩至 5 步。**

**改造后套餐详情页页面结构（从上至下）：**
```
┌──────────────────────────────────────────┐
│  ← 返回     🇯🇵 日本 7天 1GB eSIM         │
├──────────────────────────────────────────┤
│  套餐核心信息区（不变）                   │
│    运营商 · 覆盖范围 · 速度 · 有效期      │
│    质量指标（信号 89% · 网速 76% · 激活 84%）│
├──────────────────────────────────────────┤
│  天数/流量/数量 步进选择区（2.2 节已定义） │
├──────────────────────────────────────────┤
│  目的地旅行信息 折叠区（可选展开）        │
├──────────────────────────────────────────┤
│  ── 购买区（原购买确认页内容内联）──      │
│  💳 Visa **** 4242       [更换卡 >]      │
│  🏷 优惠码     [输入优惠码]   [确认]      │
│                                          │
│  小计    $9.99                           │
│  折扣    -$1.00（优惠码：TIGER10）       │
│  实付    $8.99 USD ↕                    │  ← 货币切换
├──────────────────────────────────────────┤
│  [        立即支付 $8.99        ]        │  ← 吸底主 CTA
└──────────────────────────────────────────┘
```

**交互规范：**
- 「立即支付」CTA 始终吸底，随页面内容滚动固定在底部
- 选择支付方式：点击「更换卡 >」展开 Bottom Sheet（已保存卡列表 + Apple Pay/Google Pay + 添加新卡），选完后 Bottom Sheet 关闭，回到当前页
- 优惠码输入：点击输入框后键盘弹起，确认后收起，总价实时更新
- 货币切换「↕」：Bottom Sheet 展示货币列表，选择后总价实时换算
- 购买数量 > 1 张时：吸底区展示「×N 张，共 $XX.XX」

**移除的独立购买确认页：**
- 此优化实施后，原「购买确认页」（Module 3）不再作为独立导航层级
- Module 3 的各功能组件（折扣码 3.1、多币种 3.2、已保存卡 3.3、Apple Pay 3.4）均内联于此区域

### 模块 4：支付流程（Payment）

#### 4.1 支付宝偶发无法到达付款页修复（P0）

**问题：** 点击「支付宝支付」后，偶发性跳转失败——未能成功唤起支付宝 App，或唤起后无法到达付款页，导致用户停留在空白过渡态，无任何提示或重试入口。

**根因方向：** 支付宝 SDK 唤起链路偶发超时 / 回调未正确处理，需前端增加超时检测 + 降级提示。

> 超时设计说明：8 秒是「SDK 唤起检测超时」——即检测支付宝 App 是否被成功拉起，**不是**等待用户在支付宝内完成支付的时间。一旦支付宝 App 成功打开，用户完成支付的结果通过服务端回调（Webhook）异步通知，前端监听回调更新状态，无需另设等待超时。

**需新增的过渡页面设计：**

**「支付中」等待页（正常态）：**
```
        [Tiger eSIM Logo]
        
        正在跳转支付宝...
        请在支付宝完成支付后返回
        
        [⏳ 动效 loading 圈]
        
        ──────────────────
        支付遇到问题？
        [重新发起支付]   [联系客服]
```

**跳转失败降级提示（异常态，超时 8s 后触发）：**
```
        [⚠️ 图标]
        
        支付宝跳转失败
        可能是网络不稳定，请重试
        
        [重新发起支付]
        [换个方式支付]   [联系客服]
```

- 全屏白底，品牌 logo 居中
- Loading 动效：橙色圆弧旋转
- 8 秒超时后自动切换为失败态，不让用户停留在空白中
- 失败态提供「重新发起支付」「换个方式支付」双出路

**Figma 需出图：**
- [ ] 支付跳转等待页（正常态）
- [ ] 跳转失败降级页（8s 超时态，含重试 + 换付款方式入口）

#### 4.2 银行卡支付失败 UI（P0）

**触发场景与对应文案：**

| 失败原因 | 用户提示文案 |
|---------|------------|
| 余额不足 | 「你的卡余额不足，请换一张卡或充值后重试」 |

**失败页面设计（含内联换卡，P1 优化重点）：**

```
        [❌ 图标]

        支付未成功
        [具体原因文字，对应上表]

        ──────── 换一张卡重试 ────────
        ○  💳 Visa **** 4242      ← 当前失败的卡（不可选）
        ●  💳 Mastercard **** 8888  ← 其他已保存卡（可直接选择）
        ○  💳 UnionPay **** 3311
           [+ 使用新卡]
        ──────────────────────────

        [    用选中的卡重新支付    ]   ← 主 CTA，橙色

        [联系客服]

        📧 如已扣款但显示失败，请联系客服
           我们将在 24 小时内核实处理
```

**交互规范（P1 优化：消除跳转，内联换卡）：**
- 失败页直接展示用户已保存的其他银行卡列表（最多显示 3 张，超出显示「查看更多」）
- 失败的卡默认标记为「不可选」（灰色，加锁图标），其余卡可直接点击选中

### 模块 15：套餐使用质量反馈（Plan Quality Feedback）—— P2

> **设计决策：不做公开文字评价。** Tiger 底层信号质量依赖运营商，属于黑盒，无法自主保障。公开差评会永久损害转化率，且用户无处切换导致无从「用脚投票」，评价失去核心价值。替代方案：在套餐详情页展示**聚合质量指标**（用户不可见原始内容），内部沉淀结构化反馈用于选品优化。

**位置：** 套餐详情页底部区块

**展示设计（用户侧可见）：**

```
─────────── 真实用户使用数据 ───────────────

  基于 2,341 次使用反馈

  信号覆盖    ████████████░░  89%  稳定
  网络速度    ████████░░░░░░  76%  满意
  激活体验    ██████████░░░░  84%  顺畅

  ────────────────────────────────────────
  ℹ️ 数据来自购买并激活本套餐的用户，每 7 天更新
```

- **无文字评价展示**：只展示三个维度的百分比，不展示任何个人评论
- **无星级评分**：避免「平均分」在差评积累后拉低整体印象
- **空态（首批数据不足时）**：显示「数据收集中，首批反馈将在 30 天后呈现」，不展示任何占位内容

**反馈收集（内部，不对用户展示原始内容）：**

触发时机：套餐到期后第 3 天，推送通知「帮助我们改进，你的反馈让下一位旅行者更顺利」→ Deep Link `tiger://plans/feedback?plan_id=XXX`

反馈弹窗（用户填写）：
```

  这次出行网络体验如何？

  激活体验      [👍 顺畅]  [👎 遇到问题]

  ──────────────────────────────────────
  遇到了什么问题？（选填，仅供内部改进）
  [___________________________________]

  [提交，获得 100 Tiger 币]          [跳过]
```

- 三个维度 👍/👎 单选（必填其一）
- 文字框选填，字数上限 200 字，**仅运营团队可见，不对外展示**
- 提交后即刻发放 100 Tiger 币，Toast 提示「感谢反馈！100 Tiger 币已到账」
- 每个套餐订单只触发一次，跳过后不再重复推送

**内部数据用途：**
- 各套餐维度得分低于阈值（< 70%）→ 运营介入，考虑更换底层运营商或下架该产品

### 模块 17：智能推送通知策略（Smart Notifications）—— P1

**现状：** 推送策略混乱或缺失，通知点击后仅打开 App 首页，没有 Deep Link 到具体场景，用户需要重新自行操作

#### 17.2 Deep Link 断层修复规格（P0 优化重点）

**核心问题：** 通知点击必须携带上下文参数，让用户零操作完成目标动作，而不是让用户自己重新搜索

**Deep Link Scheme 规格：**
```
tiger://plans/addon?plan_id=XXX&country=JP        ← 流量告急 → 加购页（预筛选国家）
tiger://orders/renew?plan_id=XXX                   ← 套餐到期 → 续订确认页（预填套餐）
tiger://activation/status?order_id=XXX&step=3     ← eSIM 就绪 → 激活进度页（定位步骤3）
tiger://plans/list?country=JP                      ← 复购 → 套餐列表（预筛选国家）
tiger://loyalty/redeem                             ← 积分过期 → 兑换页
tiger://plans/feedback?plan_id=XXX                ← 质量反馈提醒 → 内部反馈弹窗（不公开展示）
tiger://referral/history                           ← 邀请奖励 → 邀请记录
```

**前端路由要求：**
- App 冷启动（后台未运行）时点击通知：App 初始化完成后直接导航到目标页，不经过首页
- App 热启动（在后台）时点击通知：立即导航到目标页
- 落地页参数无效（如 plan_id 已失效）：回退到对应的列表页（如退回套餐列表），不能直接报错或空白
